## Design Context

### Users

**Who**: Vietnamese-speaking individuals focused on fitness, nutrition, and meal planning. They range from casual health-conscious eaters to serious gym-goers tracking macros.

**Context**: Users interact primarily on mobile (Android via Capacitor), often while cooking, grocery shopping, or planning meals for the week. They need quick, glanceable data — calorie totals, macro breakdowns, meal plans — with minimal friction.

**Job to be done**: Plan nutritious meals that align with personal fitness goals (cut, maintain, or bulk), track daily macro intake, and make informed food choices — all in Vietnamese, offline-first, with optional AI-powered food photo analysis.

### Brand Personality

**3-word personality**: Clean · Smart · Motivating

**Voice & tone**: Approachable and encouraging, never clinical or judgmental. Uses concise Vietnamese language — conversational but informative. Numbers and data are presented with clarity, not overwhelm. The app feels like a knowledgeable friend who helps you eat better, not a strict dietitian.

**Emotional goals**: Users should feel **calm and organized** — confident that their nutrition is on track, with a sense of control over their health journey. The interface should reduce anxiety around meal planning, not add to it.

### Aesthetic Direction

**Visual tone**: Minimal and clean with purposeful data density. The design follows Apple Health and MyFitnessPal's calm data display philosophy — information-rich but never cluttered. White space is intentional. Color is used sparingly for meaning (macro categories, status, AI features).

**Theme**: Emerald-anchored (#10b981) — a fresh, health-associated green that avoids the clinical feel of blue or the urgency of red. Slate neutrals provide a sophisticated backdrop. Full light + dark mode support with OKLCH perceptual uniformity.

**References**:

- **Apple Health**: Calm, card-based data display with clear hierarchy
- **MyFitnessPal**: Nutrition tracking patterns, macro donut charts
- **shadcn/ui base-nova**: Modern, accessible component baseline

**Anti-references**:

- No gamification (badges, streaks, leaderboards) — this is a planning tool, not a game
- Not overly clinical/medical — avoid hospital-blue palettes or warning-heavy UI
- No neon/vibrant gradients — the app should feel grounded, not flashy
- No dark patterns — never guilt users about food choices

### Design Principles

1. **Clarity over decoration**: Every visual element must earn its place. Prefer whitespace and hierarchy over ornament. If a user can't understand a screen in 3 seconds, simplify it.

2. **Data with empathy**: Nutrition numbers can be stressful. Present data as progress, not judgment. Use warm emerald tones for positive states; reserve red/rose only for true destructive actions, never for "you ate too much."

3. **Mobile-first, touch-native**: Design for thumbs on a 6" screen. Minimum 44px touch targets. Bottom-anchored actions. Swipe-friendly interactions. Safe area awareness for notches and home indicators.

4. **Semantic everything**: Colors, spacing, typography — always via tokens, never raw values. Every color must have meaning (macro type, status, feature domain). Dark mode isn't an afterthought — it's a first-class citizen with intentional OKLCH pairs.

5. **Accessible by default**: WCAG 2.1 AA minimum. Focus rings visible. Screen reader support. Reduced motion respected. Contrast ratios enforced. Vietnamese text rendering optimized (diacritics, line-height).

**Rules derived from audit:**

- ✅ **DO**: Use `--primary` for buttons (white text on emerald bg), icons (≥16px), large headings (≥18px bold / ≥24px regular)
- ❌ **DON'T**: Use `--primary` as foreground color on white for body text (<18px). Use `--primary-emphasis` (Emerald-700, higher contrast) instead
- ✅ **DO**: Use `--muted-foreground` (Slate-600) for all secondary/caption text — it passes AA at 7.58:1
- ❌ **DON'T**: Use `--macro-fat` (Amber-600) or `--destructive` (Red-500) as small text on white — large text / icons only
- 🎯 **TARGET**: Migrate primary small text to Emerald-700 (`--primary-emphasis`) for full AA compliance

#### Domain Colors (Semantic, Not Decorative)

| Domain  | Token             | Color       | Context Rule                                                        |
| ------- | ----------------- | ----------- | ------------------------------------------------------------------- |
| Protein | `--macro-protein` | Emerald-500 | **Nutrition context only** — macro charts, nutrition cards          |
| Fat     | `--macro-fat`     | Amber-600   | **Nutrition context only** — large text/icons (fails AA small text) |

| Deficit | `--color-rose`    | Rose-500    | Calorie deficit warnings                                            |

**Disambiguation — when colors overlap:**

Emerald appears as both `--primary` (UI) and `--macro-protein` (data). Amber appears as `--status-warning`, `--macro-fat`, and `--color-energy`.

Decision tree for developers:

```
Q: Am I coloring a nutrition data value (macro chart, nutrition card)?
├─ YES → Use --macro-* tokens (protein/fat/carbs/fiber)
└─ NO → Q: Am I coloring a UI control (button, badge, link)?
         ├─ YES → Use --primary, --destructive, --color-ai
         └─ NO → Q: Am I showing a status (success/warning/error)?
                  ├─ YES → Use --status-* tokens
                  └─ NO → Use --foreground or --muted-foreground
```

**Banned color families**: gray-\*, zinc-\*, green-\* (use emerald), violet-\*, teal-\*, yellow-\* (use amber).

### 2. Typography

#### Type Scale (Tailwind v4 Defaults — No Custom Override)

> ⚠️ **Source of truth**: Tailwind v4 defaults. `src/styles/tokens.css` defines a different scale (`--text-xs: 10px`) but **is NOT imported** — it's dead code. Sizes below are the REAL rendered values.

| Level         | Class                           | Size             | Weight | Line-height               | Usage                                      |
| ------------- | ------------------------------- | ---------------- | ------ | ------------------------- | ------------------------------------------ |
| Display       | `text-3xl font-semibold`        | 30px (1.875rem)  | 600    | `leading-tight` (1.25)    | Hero numbers (total calories)              |
| Page title    | `text-2xl font-semibold`        | 24px (1.5rem)    | 600    | `leading-tight`           | Top-level headings                         |
| Section title | `text-lg font-semibold`         | 18px (1.125rem)  | 600    | `leading-snug` (1.375)    | Card headers, section titles               |
| Subsection    | `text-base font-semibold`       | 16px (1rem)      | 600    | `leading-normal` (1.5)    | Sub-headers, labels                        |
| Body          | `text-sm`                       | 14px (0.875rem)  | 400    | `leading-relaxed` (1.625) | Primary content (416 refs — de facto body) |
| Caption       | `text-xs text-muted-foreground` | 12px (0.75rem)   | 400    | `leading-relaxed`         | Metadata, helpers (381 refs)               |
| Micro         | `text-[10px]`                   | 10px (arbitrary) | 700    | `leading-tight`           | Badge labels, unit suffixes only (52 refs) |
| Numeric       | `tabular-nums`                  | varies           | varies | —                         | All number displays                        |

**Font**: Geist Variable (sans-serif). Supports Vietnamese diacritics. System fallback: `ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif`.

> **Known heading inversion**: H1 (Page Title `text-lg` = 18px) is SMALLER than H2 (Screen Title `text-2xl` = 24px) in some components. 🎯 Target: standardize H1 ≥ H2.

#### Font Weight — Current vs Target

| Weight | Class           | Current usage | 🎯 Target | Rule                                                                     |
| ------ | --------------- | :-----------: | :-------: | ------------------------------------------------------------------------ |
| 700    | `font-bold`     | **261 refs**  |    ~50    | 🎯 Migrate to semibold. Only for display numbers (≥24px) and stat values |
| 600    | `font-semibold` |   175 refs    |   ~350    | Headings, prominent labels, button text                                  |
| 500    | `font-medium`   |   294 refs    |   ~300    | Form labels, secondary emphasis, nav items                               |
| 400    | (default)       |   majority    | majority  | Body text, descriptions                                                  |

**Rule**: `font-bold` (700) creates visual heaviness on mobile. Prefer `font-semibold` (600) for headings and `font-medium` (500) for labels. Reserve `font-bold` only for hero numbers at `text-2xl`+.

#### Vietnamese Typography Rules

Vietnamese diacritics (ắ, ặ, ỡ, ữ, ể) extend above and below the Latin baseline. Without sufficient line-height, diacritics clip or collide.

| Rule                    | Specification                                | Rationale                                                                             |
| ----------------------- | -------------------------------------------- | ------------------------------------------------------------------------------------- |
| **Body line-height**    | `leading-relaxed` (1.625) minimum            | Diacritics need vertical room                                                         |
| **Caption line-height** | `leading-relaxed` minimum                    | Even at 12px (`text-xs`), Vietnamese needs vertical space                             |
| **Heading line-height** | `leading-snug` (1.375) acceptable            | Larger sizes have proportionally more room                                            |
| **Min readable size**   | `text-xs` (12px) absolute minimum            | Smaller breaks diacritics on 420dpi mobile. `text-[10px]` only for decorative badges. |
| **Text truncation**     | `line-clamp-2` for dish/ingredient names     | Vietnamese names can be long ("Bông cải xanh xào tỏi ớt với nấm hương")               |
| **Ellipsis**            | `truncate` (single-line) for nav labels only | Multi-line clamp preferred elsewhere                                                  |
| **Number rendering**    | `font-variant-numeric: tabular-nums`         | Aligned columns in nutrition tables                                                   |

**Do / Don't:**

- ✅ **DO**: `<p className="text-sm leading-relaxed">Mô tả món ăn...</p>`
- ❌ **DON'T**: `<p className="text-sm leading-none">Mô tả...</p>` — diacritics will clip
- ✅ **DO**: `<h3 className="text-lg font-semibold leading-snug line-clamp-2">Tên rất dài...</h3>`
- ❌ **DON'T**: `<span className="text-[9px]">...</span>` — below minimum readable size

> ⚠️ **Active violations**: 5 instances of `leading-none` on Vietnamese text content:
> `label.tsx:11`, `dialog.tsx:106`, `GoalDetailPage.tsx:83`, `HealthProfileDetailPage.tsx:79`, `TrainingProfileSection.tsx:141`
> 🎯 **Fix**: Replace with `leading-snug` minimum.

### 6. Motion & Choreography

#### Keyframes

| Animation | Duration | Easing     | Transform                | Usage              |
| --------- | -------- | ---------- | ------------------------ | ------------------ |
| `fadeIn`  | 200ms    | `ease-out` | opacity 0→1              | General entrance   |
| `slideUp` | 300ms    | `ease-out` | translateY(8px)→0 + fade | Cards, list items  |
| `scaleIn` | 150ms    | `ease-out` | scale(0.95)→1 + fade     | Popovers, tooltips |

#### Easing Tokens

| Token           | Curve                               | Usage                            |
| --------------- | ----------------------------------- | -------------------------------- |
| `--ease-enter`  | `cubic-bezier(0, 0, 0.2, 1)`        | Elements entering viewport       |
| `--ease-exit`   | `cubic-bezier(0.4, 0, 1, 1)`        | Elements leaving viewport        |
| `--ease-spring` | `cubic-bezier(0.34, 1.56, 0.64, 1)` | Bouncy micro-feedback            |
| `--ease-linear` | `linear`                            | Progress bars, continuous motion |

#### Choreography Rules

| Interaction              | Animation                    | Detail                                                 |
| ------------------------ | ---------------------------- | ------------------------------------------------------ |
| **Button press**         | `active:scale-[0.98]`        | Subtle scale-down on tap; feels physical               |
| **Button release**       | `transition-all`             | Spring back via CSS transition                         |
| **Sheet enter** (bottom) | slide-up 200ms + fade        | `data-starting-style:translate-y-[2.5rem]`             |
| **Sheet exit**           | slide-down 200ms + fade      | Reverse of enter                                       |
| **Modal overlay**        | fade 150ms                   | `bg-black/10` with `backdrop-blur-xs`                  |
| **List stagger**         | 30ms per item, max tier 5    | `.animate-stagger-1` through `-5` (cap at 150ms total) |
| **Tab switch**           | No transition                | Instant swap — tabs feel "always loaded"               |
| **Page stack push**      | Slide left (overlay pattern) | `PageStackOverlay` renders on top                      |
| **Card hover**           | `shadow-sm → shadow-md`      | `transition-shadow` on hover                           |

#### Reduced Motion

```css
@media (prefers-reduced-motion: reduce) {
  .animate-fade-in,
  .animate-slide-up,
  .animate-scale-in {
    animation: none !important;
    /* Elements appear instantly — no delay, no movement */
  }
}
```

**Rule**: When `prefers-reduced-motion: reduce`, ALL slide/scale animations are disabled. Only opacity fades may remain (they don't trigger vestibular issues). `transition-all` on buttons should be scoped to `transition-colors` under reduced motion.

### 9. Interaction States

Every interactive element must handle **all 6 states**. This is not optional.

#### State Matrix

| State              | Visual Treatment                                                            | Accessibility                                  |
| ------------------ | --------------------------------------------------------------------------- | ---------------------------------------------- |
| **Default**        | Base styling                                                                | —                                              |
| **Hover**          | `hover:bg-primary/80` or `hover:shadow-md`                                  | —                                              |
| **Active/Pressed** | `active:scale-[0.98]` + `active:translate-y-px`                             | —                                              |
| **Focus-visible**  | `focus-visible:border-ring focus-visible:ring-3 focus-visible:ring-ring/50` | Keyboard users see focus ring                  |
| **Disabled**       | `disabled:pointer-events-none disabled:opacity-50`                          | `aria-disabled` or native `disabled` attribute |
| **Error/Invalid**  | `aria-invalid:border-destructive aria-invalid:ring-destructive/20`          | `aria-invalid="true"` + `aria-describedby`     |

#### Loading States

| Pattern                | Component                             | Usage                                |
| ---------------------- | ------------------------------------- | ------------------------------------ |
| **Skeleton shimmer**   | `<Skeleton className="h-4 w-full" />` | Content placeholder while data loads |
| **Inline spinner**     | SVG spinner inside button             | Button actions (save, submit)        |
| **Progressive reveal** | Skeleton → fade-in content            | Lists, cards, charts                 |

Skeleton implementation: `bg-muted animate-pulse rounded-md`.

**Do / Don't:**

- ✅ **DO**: Show skeleton matching the layout shape of final content
- ❌ **DON'T**: Show a centered spinner for page-level loading — use skeletons that match the page layout
- ✅ **DO**: Disable submit button + show spinner while saving
- ❌ **DON'T**: Leave button enabled during async operations

#### Error States

| Element           | Treatment                              | Example                                                      |
| ----------------- | -------------------------------------- | ------------------------------------------------------------ |
| **Input field**   | Red border ring + error message below  | `aria-invalid:border-destructive`                            |
| **Error message** | `text-destructive text-xs` below input | `<p role="alert" className="text-destructive mt-1 text-xs">` |
| **Toast**         | Rose-tinted toast at top               | `--toast-error` background                                   |
| **Inline alert**  | Card with destructive border           | For non-field errors                                         |

**Rule**: Error messages always use `role="alert"` for screen readers. Always pair visual error (red border) with text explanation — color alone is insufficient (color-blind users).

#### Empty States

Pattern (from `EmptyState` component):

```
┌─────────────────────────────────────┐
│                                     │
│     [Icon in emerald circle]        │
│                                     │
│     "Chưa có {entity} nào"         │
│     "Tạo {entity} đầu tiên..."     │
│                                     │
│     [ + Tạo {entity} ]             │  ← CTA button (only if not searching)
│                                     │
└─────────────────────────────────────┘
```

- Wrapper: `bg-card border-border rounded-2xl border border-dashed p-8 text-center`
- Icon container: `bg-primary-subtle mx-auto h-16 w-16 rounded-full` (emerald circle)
- Title: `text-lg font-bold text-foreground`
- Description: `text-sm text-muted-foreground`
- CTA: Primary button with Plus icon

**Tone**: Encouraging, never blame the user. "Chưa có món ăn nào" (neutral), NOT "Bạn chưa tạo gì cả" (blaming).

#### Disabled States

- Visual: `opacity-50` — reduces all colors to 50% opacity
- Behavior: `pointer-events-none` — ignores all clicks/taps
- Cursor: `cursor-not-allowed` (via peer selector on parent for label)
- Screen reader: Native `disabled` attribute preferred over `aria-disabled`

**Do / Don't:**

- ✅ **DO**: `<Button disabled>Không thể xóa</Button>` — native disabled blocks events
- ❌ **DON'T**: `<Button aria-disabled="true" onClick={...}>` — click handler still fires!
- ✅ **DO**: Explain WHY something is disabled (tooltip or adjacent text)
- ❌ **DON'T**: Just gray out a button with no explanation

### 10. Responsive Strategy

#### Breakpoints

| Breakpoint | Width   | Target                       | Usage in codebase  |
| ---------- | ------- | ---------------------------- | :----------------: |
| Default    | 0–639px | Mobile phones (primary)      | **~85% of styles** |
| `sm:`      | ≥640px  | Large phones / small tablets |    **74 refs**     |
| `md:`      | ≥768px  | Tablets                      |       8 refs       |
| `lg:`      | ≥1024px | Desktop                      |       3 refs       |

This is a **mobile-dominant app**. Desktop is secondary. Design mobile first, then enhance for `sm:`.

#### Layout Adaptation Rules

| Pattern              | Mobile (default)        | Tablet+ (sm:)                      |
| -------------------- | ----------------------- | ---------------------------------- |
| **Macro grid**       | `grid-cols-2`           | `grid-cols-4`                      |
| **Card padding**     | `p-4`                   | `p-6`                              |
| **Page padding**     | `px-4 py-6`             | `px-6 py-8`                        |
| **Bottom clearance** | `pb-28` (nav)           | `pb-8` (no nav)                    |
| **Navigation**       | Bottom tab bar          | Desktop sidebar (hidden on mobile) |
| **Modals**           | Full-width bottom sheet | Centered dialog with max-width     |
| **Tables**           | Horizontal scroll       | Full table visible                 |

#### Touch Targets

| Element           | Min size                      | Class                                      |
| ----------------- | ----------------------------- | ------------------------------------------ |
| Buttons           | 44×44px                       | `min-h-11` (44px) or `h-8` + `p-2` padding |
| Icon buttons      | 32×32px min, 44×44px tap area | `size-8` with padding                      |
| Nav items         | 48×48px                       | `min-h-12`                                 |
| List action icons | 44×44px tap area              | Parent padding provides target size        |

**Rule**: If the visual element is smaller than 44px, ensure the tappable area (via padding) reaches 44px minimum.

### 11. Component Patterns

#### Buttons

Variants via CVA (Class Variance Authority):

| Variant       | Styling                              | Usage                          |
| ------------- | ------------------------------------ | ------------------------------ |
| `default`     | `bg-primary text-primary-foreground` | Primary actions (Save, Create) |
| `outline`     | `border-border bg-background`        | Secondary actions              |
| `ghost`       | `hover:bg-muted` (transparent)       | Tertiary, inline actions       |
| `destructive` | `bg-destructive/10 text-destructive` | Delete, remove                 |
| `success`     | `bg-primary text-white`              | Positive confirmations         |
| `warning`     | `bg-amber-500 text-amber-950`        | Caution actions                |
| `ai`          | `bg-indigo-500 text-white`           | AI-powered features            |
| `link`        | `text-primary underline`             | Text-like navigation           |

Sizes: `xs` (h-6) → `sm` (h-7) → `default` (h-8) → `lg` (h-9) → `icon` (size-8).

#### Cards

```
bg-card rounded-xl ring-1 ring-foreground/10
```

- Size `default`: `gap-4 py-4`
- Size `sm`: `gap-3 py-3`
- Hover (interactive cards): `hover:shadow-md transition-shadow`
- Structure: `CardHeader > (CardTitle + CardDescription + CardAction)`, `CardContent`, `CardFooter`

#### Forms

- Framework: React Hook Form + Zod
- Wrapper: `<FormField label error required htmlFor>` — auto-connects label to input, shows error with `role="alert"`
- Radio groups: `<RadioPills>` — emerald active, muted inactive, 44px min-height
- Numeric inputs: `font-variant-numeric: tabular-nums`, `type="text"` with pattern validation (NOT `type="number"` — avoids mobile keyboard issues)
- Multi-step: Validate ONLY current step fields with `form.trigger([...fields])`, NEVER `form.trigger()` without args
- Error display: Inline `text-destructive text-xs` below each field, `role="alert"`, `aria-describedby` linking

#### Modals & Sheets

- Overlay: `ModalBackdrop` with scroll-lock (reference-counted depth for nested modals)
- Z-index: `z-50`+ (sheets), `z-[60]` (fitness modals)
- Close: Escape key (stacked — only topmost modal responds), backdrop click
- Focus trap: Managed by base-ui
- Sheet animation: slide-up 200ms from bottom, `backdrop-blur-xs` on overlay
- **Do**: Keep modal content scannable — max 3 actions in footer
- **Don't**: Stack more than 2 modals deep (e.g., modal + unsaved changes dialog = max)

#### Lists

- Card-based items with `flex justify-between items-start`
- Actions right-aligned: icon buttons in `flex gap-2`
- Hover: `hover:shadow-md transition-shadow`
- Empty state: `EmptyState` component when list is empty or search returns nothing

#### Nutrition Displays

- Grid: `grid-cols-2` (mobile) → `grid-cols-4` (sm:)
- Values: colored by macro (`--macro-protein`, `--macro-fat`, `--macro-carbs`)
- Hierarchy: Value (large, bold, colored) > Label (caption, muted) > Percentage (caption)
- Numbers: `tabular-nums`, `Math.round()` — no decimals for kcal/g
- Progress bars: `bg-muted h-1.5 overflow-hidden rounded-full` with colored fill
- Overflow handling: When eaten > target, bar clamps at 100% visually, text shows "Vượt: X kcal"

### 12. Content & Data Display Guidelines

#### Number Formatting

| Data type         | Format                | Example | Rule                         |
| ----------------- | --------------------- | ------- | ---------------------------- |
| Calories          | Integer, no separator | `2091`  | `Math.round()`, unit: `kcal` |
| Protein/Fat/Carbs | Integer + `g`         | `170g`  | `Math.round()`, no decimal   |
| Weight            | 1 decimal             | `75.5`  | Unit: `kg`                   |
| Percentage        | Integer + `%`         | `31%`   | Round to nearest integer     |
| BMR/TDEE          | Integer               | `1704`  | No separator, no decimal     |

**Rule**: All nutrition numbers use `Math.round()` — never show decimals for kcal or grams. Use `tabular-nums` for aligned columns.

#### Copy Tone (Vietnamese)

| Context          | ✅ Do                                    | ❌ Don't                      |
| ---------------- | ---------------------------------------- | ----------------------------- |
| **Over budget**  | "Bạn đang vượt 200 kcal so với mục tiêu" | "Bạn ăn quá nhiều!"           |
| **Under budget** | "Bạn mới nạp 1200 kcal — hãy nạp thêm!"  | "Thiếu 800 kcal — nguy hiểm!" |
| **Empty state**  | "Chưa có món ăn nào"                     | "Bạn chưa tạo gì cả"          |
| **Success**      | "Đã lưu thành công!"                     | "OK"                          |
| **Error**        | "Không thể lưu. Vui lòng thử lại."       | "Lỗi!"                        |

**Principle**: State facts neutrally → suggest action positively → never blame the user.

#### Unit Display

| Unit           | Display            | Context                          |
| -------------- | ------------------ | -------------------------------- |
| Calories       | `kcal` (lowercase) | Always after number: "2091 kcal" |
| Weight (macro) | `g` (lowercase)    | Appended: "170g" (no space)      |
| Body weight    | `kg`               | With space: "75.5 kg"            |
| Height         | `cm`               | With space: "175 cm"             |

#### Progress Bar Overflow Behavior

```
Eaten ≤ Target:  [████████░░░░] Còn lại: 764 kcal
Eaten > Target:  [████████████] Vượt: 209 kcal   ← bar capped at 100%, text changes
Eaten = 0:       [░░░░░░░░░░░░] 2091 kcal còn lại
```

The bar NEVER exceeds its container. Overflow communicated via text only.

#### Text Truncation Rules

| Context               | Strategy         | Class                   |
| --------------------- | ---------------- | ----------------------- |
| Dish name in card     | 2-line clamp     | `line-clamp-2`          |
| Dish name in list row | Single truncate  | `truncate`              |
| Nav label             | Truncate at 72px | `max-w-[72px] truncate` |
| Ingredient name       | 2-line clamp     | `line-clamp-2`          |
| Description text      | 3-line clamp     | `line-clamp-3`          |

**Rule**: Always provide full text via `title` attribute or tooltip when truncating.

#### Touch Target vs Button Default Mismatch

The `Button` component default size is `h-8` (32px). The design principle says minimum touch target = 44px.

| Button size | Visible height | Touch-target compliant?  |
| ----------- | :------------: | :----------------------: |
| `xs`        |   h-6 (24px)   |     ❌ Needs padding     |
| `sm`        |   h-7 (28px)   |     ❌ Needs padding     |
| `default`   |   h-8 (32px)   |     ❌ Needs padding     |
| `lg`        |   h-9 (36px)   | ⚠️ Close but still <44px |
| `icon`      | size-8 (32px)  |     ❌ Needs padding     |

**Resolution**: The 44px rule applies to the **tappable area** (including padding), not the visible button height. Ensure parent containers or spacing provide at least 44px tappable area. For standalone buttons (modals, forms), use `min-h-11` (44px) wrapper or add `py-2` padding.

> 🎯 **Target**: Add `min-h-11` to Button's default variant for standalone usage, or document explicit cases where smaller is acceptable (inline actions within lists where row padding provides the target).

### 17. Additional Design Rules (v3.1)

#### Rule 1: Button↔Modal Color Pairing

**Requirement**: When a button triggers a modal, the modal's header/accent color MUST match the button's variant color. This ensures visual continuity between the trigger and the resulting dialog.
**Pattern**:

```tsx
// Amber trigger → amber modal header
<Button variant="warning" onClick={openModal}>Lưu mẫu</Button>
<ModalHeader className="bg-status-warning/10 text-status-warning">Lưu mẫu bài tập</ModalHeader>
```

**Anti-pattern**:

```tsx
// ❌ Amber button opens emerald modal — color mismatch breaks visual flow
<Button variant="warning" onClick={openModal}>Lưu mẫu</Button>
<ModalHeader className="bg-emerald-500/10 text-emerald-700">Lưu mẫu bài tập</ModalHeader>
```

#### Rule 2: Modal Max-Width & Layout

**Requirement**: Mobile modals = full-width with `max-w-[calc(100vw-2rem)]`. Desktop = centered with `sm:max-w-[500px]`. Never use `max-w-lg` without a responsive prefix — it clips content on small screens.
**Pattern**:

```tsx
<DialogContent className="max-w-[calc(100vw-2rem)] sm:max-w-[500px]">{/* modal body */}</DialogContent>
```

**Anti-pattern**:

```tsx
// ❌ max-w-lg without responsive prefix — clips on 320px screens
<DialogContent className="max-w-lg">{/* modal body */}</DialogContent>
```

#### Rule 3: Active Tab Indicator

**Requirement**: Active tab = `border-b-2 border-primary text-foreground font-semibold`. Inactive = `text-muted-foreground`. Never use raw color classes (e.g., `text-blue-600`, `border-emerald-500`) for tab indicators — always use semantic tokens.
**Pattern**:

```tsx
<button
  className={cn(
    'pb-2 text-sm transition-colors',
    isActive ? 'border-primary text-foreground border-b-2 font-semibold' : 'text-muted-foreground',
  )}
>
  {label}
</button>
```

**Anti-pattern**:

```tsx
// ❌ Raw blue for active tab — breaks theming
<button className={isActive ? 'border-b-2 border-blue-600 text-blue-600' : 'text-gray-500'}>{label}</button>
```

#### Rule 4: Required Field Indicator

**Requirement**: Required form fields MUST show a red asterisk after the label. Never rely solely on validation error messages to communicate that a field is required — the asterisk provides proactive visual cue.
**Pattern**:

```tsx
<Label htmlFor="name">
  Tên món ăn
  <span className="text-destructive ml-1">*</span>
</Label>
```

**Anti-pattern**:

```tsx
// ❌ No visual indicator — user only discovers field is required after submit fails
<Label htmlFor="name">Tên món ăn</Label>
```

#### Rule 5: Badge Component Styling

**Requirement**: Badges use `rounded-full text-[10px] font-semibold px-2 py-0.5`. Use semantic color tokens for each severity level. Never use raw Tailwind color classes for badge backgrounds.
**Pattern**:

```tsx
// Info badge
<span className="rounded-full text-[10px] font-semibold px-2 py-0.5 bg-primary/10 text-primary">Mới</span>
// Error badge
<span className="rounded-full text-[10px] font-semibold px-2 py-0.5 bg-destructive/10 text-destructive">Lỗi</span>
// Warning badge
<span className="rounded-full text-[10px] font-semibold px-2 py-0.5 bg-status-warning/10 text-status-warning">Cảnh báo</span>
```

**Anti-pattern**:

```tsx
// ❌ Raw colors + inconsistent sizing
<span className="rounded bg-red-100 px-1 text-xs text-red-800">Lỗi</span>
```

#### Rule 6: Input Focus State Rule

**Requirement**: ALL interactive inputs MUST have visible focus and error states using ring utilities. This ensures accessibility (WCAG 2.4.7 Focus Visible) and consistent visual feedback across the app.
**Pattern**:

```tsx
<Input className="focus-visible:border-ring focus-visible:ring-ring/50 aria-invalid:border-destructive aria-invalid:ring-destructive/20 focus-visible:ring-3" />
```

**Anti-pattern**:

```tsx
// ❌ No focus ring — fails WCAG focus-visible requirement
<Input className="border border-input" />
// ❌ Outline-none removes all focus indicators
<Input className="outline-none border border-input" />
```

#### Rule 7: Back Button Pattern

**Requirement**: Back buttons use ghost variant, icon size, with `ChevronLeft`. Position top-left. Always include `aria-label` for screen readers.
**Pattern**:

```tsx
<Button variant="ghost" size="icon" aria-label="Quay lại" onClick={goBack}>
  <ChevronLeft className="h-5 w-5" />
</Button>
```

**Anti-pattern**:

```tsx
// ❌ Text button for back — wastes horizontal space, inconsistent with app pattern
<Button variant="outline" onClick={goBack}>← Quay lại</Button>
// ❌ Missing aria-label — screen reader announces "button" with no context
<Button variant="ghost" size="icon" onClick={goBack}>
  <ChevronLeft className="h-5 w-5" />
</Button>
```

#### Rule 8: Skeleton Shimmer Sizing

**Requirement**: Skeleton placeholders MUST match the final content dimensions to prevent layout shift (CLS). Use `animate-pulse bg-muted rounded-lg`. Height approximates loaded content. Width = full container width.
**Pattern**:

```tsx
// Skeleton for a card that loads as 80px tall
<div className="animate-pulse bg-muted rounded-lg h-20 w-full" />
// Skeleton for a text line
<div className="animate-pulse bg-muted rounded h-4 w-3/4" />
```

**Anti-pattern**:

```tsx
// ❌ Fixed tiny skeleton for large content — causes layout shift on load
<div className="animate-pulse bg-gray-200 h-4 w-20" />
// ❌ No rounded corners — inconsistent with card design
<div className="animate-pulse bg-muted h-20 w-full" />
```

#### Rule 9: Chart & Visualization Colors

**Requirement**: Data visualizations MUST use macro tokens: `--macro-protein` (emerald), `--macro-fat` (amber), `--macro-carbs` (blue), `--macro-fiber` (green). Legends required for all charts. Inline SVG styles MUST use CSS custom properties, not hardcoded hex/rgb values.
**Pattern**:

```tsx
// Use CSS custom properties in SVG
<circle style={{ fill: 'var(--macro-protein)' }} />
<circle style={{ fill: 'var(--macro-fat)' }} />
<circle style={{ fill: 'var(--macro-carbs)' }} />
// Legend always present
<Legend items={[
  { color: 'var(--macro-protein)', label: 'Protein' },
  { color: 'var(--macro-fat)', label: 'Chất béo' },
  { color: 'var(--macro-carbs)', label: 'Carbs' },
]} />
```

**Anti-pattern**:

```tsx
// ❌ Hardcoded hex — breaks theming, impossible to maintain
<circle style={{ fill: '#10b981' }} />
// ❌ No legend — user cannot interpret chart segments
<PieChart data={macros} />
```

#### Rule 10: Icon Sizing Standard

**Requirement**: Strict icon scale: `h-4 w-4` (inline with text), `h-5 w-5` (inside buttons), `h-6 w-6` (empty states/section headers), `h-12 w-12` (hero illustrations only). BANNED sizes: `h-7`, `h-8`, `h-9`, `h-10` (non-standard gaps in the scale). Exception: data visualization SVGs may use custom dimensions.
**Pattern**:

```tsx
// Inline with text
<span className="flex items-center gap-1"><Info className="h-4 w-4" /> Ghi chú</span>
// Inside button
<Button><Plus className="h-5 w-5 mr-1" /> Thêm</Button>
// Empty state
<div className="text-center"><Inbox className="h-6 w-6 mx-auto text-muted-foreground" /></div>
```

**Anti-pattern**:

```tsx
// ❌ h-8 w-8 — non-standard size, creates visual inconsistency
<Settings className="h-8 w-8" />
// ❌ Mixing scales in same context
<div><Info className="h-4 w-4" /><Settings className="h-7 w-7" /></div>
```
